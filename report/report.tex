% ─── ABSTRACT ──────────────────────────────────────────────
\begin{tcolorbox}[colback=srmlightblue,colframe=srmblue,
  title=\textbf{Abstract},fonttitle=\small\bfseries,
  fontupper=\small,top=4pt,bottom=4pt,left=6pt,right=6pt]
This paper presents an \textbf{Autonomous SQL Agent} — an AI system that allows users to ask
business questions in plain English and automatically get the answer from a data warehouse,
without writing any SQL. The system uses eight AI agents working together: one to understand
the question, one to plan the analysis, one to look up the database structure, one to write
the SQL query (using the Qwen2.5-Coder-7B language model), one to check it is safe, one to
run it, one to find patterns in the results, and one to explain the findings. A backup
generator handles queries when the internet is unavailable. All 22 automated tests pass.
The system runs as a Streamlit web application and shows that AI agents can help
non-technical users query a data warehouse safely and reliably.

\smallskip
\textbf{Keywords:} Natural Language to SQL, AI Agents, Data Warehouse, LLM, Star Schema, Streamlit
\end{tcolorbox}

% ─── 1. INTRODUCTION ───────────────────────────────────────
\section{Introduction}

Business analysts, managers, and planners need data from company databases every day, but
most of them do not know SQL — the language used to query databases. Writing SQL for a large
data warehouse with many linked tables is especially difficult. Modern AI language models
(LLMs) can now write accurate SQL from a plain English description~\cite{brown2020gpt3}, and
multi-agent AI systems can break a complex task into smaller, manageable
steps~\cite{yao2022react}. This project combines both ideas to let any user simply type a
question and receive a correct, safe answer without touching SQL at all.

\begin{tcolorbox}[colback=srmlightblue,colframe=srmblue,
  fontupper=\small,top=3pt,bottom=3pt,left=6pt,right=6pt]
\textbf{Problem Statement.}~Given a question asked in plain English and a database schema,
automatically write the correct SQL query, run it safely, find useful patterns in the
results, and show the findings with charts and downloadable files —
\textit{without requiring any SQL knowledge from the user.}
\end{tcolorbox}

\smallskip
\textbf{Key contributions:}
\begin{enumerate}[leftmargin=1.5em,itemsep=0pt,topsep=2pt]
\item An eight-agent pipeline that converts a plain English question into SQL, with a
      smart retry mechanism (up to 2 attempts) that sends the error details back to the AI
      when the first attempt fails. A backup rule-based generator works without internet.
\item Three safety checks on every generated SQL query: a keyword filter, a code structure
      check, and a dry-run test — ensuring no harmful query ever reaches the database.
\item An automatic CSV upload feature that reads any spreadsheet and builds a proper
      database structure from it, detecting column types by their names and value counts.
\item Results exported in three formats: plain CSV, Excel with two sheets, and a PDF
      report with an embedded chart image.
\end{enumerate}

% ─── 2. SYSTEM ARCHITECTURE ────────────────────────────────
\section{System Architecture}

The system is built in four layers, shown in Figure~\ref{fig:arch}.
\textbf{Layer 1} is the Streamlit web interface where users upload files, type questions,
and download results.
\textbf{Layer 2} is the Orchestrator — the central controller that runs all agents in order,
keeps track of shared information, and handles retries.
\textbf{Layer 3} contains the eight AI agents that each do one job in the pipeline.
\textbf{Layer 4} holds the shared service modules: the database connection, schema reader,
SQL generator, chart builder, analytics engine, and export handler.

\begin{figure}[H]
\centering
\includegraphics[width=0.95\textwidth]{figs/diagram-1.png}
\caption{Four-layer system architecture. Layer~1: web UI. Layer~2: Orchestrator.
Layer~3: eight-agent pipeline. Layer~4: shared service modules.}
\label{fig:arch}
\end{figure}

\textbf{Two ways to load data.}
(1)~\textit{Built-in warehouse}: a ready-to-use retail database (described in
Section~\ref{sec:db}) that works straight away.
(2)~\textit{CSV upload}: the user uploads any spreadsheet; the system automatically builds
a proper database from it and limits all queries to that data.

\textbf{Two SQL generation modes.}
(1)~\textit{AI mode}: uses the Qwen2.5-Coder-7B language model via the HuggingFace API
when an internet connection and API key are available.
(2)~\textit{Offline mode}: a simpler rule-based generator runs automatically when the AI
service is unavailable, covering the five most common query types.

% ─── 4. DATABASE DESIGN ────────────────────────────────────
\section{Database Design}
\label{sec:db}

The built-in database follows a star schema design~\cite{kimball2013toolkit}, which
organises data into one central table of sales records surrounded by smaller lookup
tables for customers, products, regions, sales channels, and dates.
There are five lookup tables: \texttt{dim\_customer} (6,000 customers),
\texttt{dim\_product} (300 products), \texttt{dim\_region} (50 regions),
\texttt{dim\_channel} (4 channels), and \texttt{dim\_date} (1,096 dates covering 3 years).
The three main data tables are \texttt{fact\_orders} (over 100,000 orders),
\texttt{fact\_order\_items} (over 300,000 line items), and
\texttt{fact\_returns} (over 10,000 returns). Figure~\ref{fig:er} shows how all
these tables are connected.

\begin{figure}[H]
\centering
\includegraphics[width=0.88\textwidth]{figs/diagram-3.png}
\caption{Star schema diagram. The central \texttt{fact\_orders} table (blue) is linked to
five surrounding lookup tables (gray). Return and order-item tables are also connected.}
\label{fig:er}
\end{figure}

Every query run by the system is saved in a session log table
(\texttt{app\_analysis\_sessions}) that records the original question, the SQL used,
how many rows were returned, the chart type, file paths, and any warnings.

% ─── 5. IMPLEMENTATION ─────────────────────────────────────
\section{Implementation}

\subsection{SQL Generation}

When a user asks a question, the system builds a prompt for the AI model that contains:
the relevant table and column names from the database; a glossary that maps everyday words
to their column names (for example, ``revenue'' means the \texttt{total\_amount} column);
the user's question; a 4-step analysis plan; and rules for writing SQLite-style SQL.
The AI returns a JSON response containing the SQL query, a suggested chart type, and the
table names used. If the response is in an unexpected format, the system tries several
ways to extract the SQL from it.

When the AI is not available, a backup generator reads keywords from the question to
choose one of five ready-made query templates: top-N ranking, monthly trend, returns by
region, customer segmentation, and a general summary. It fills in the correct column names
from the database automatically. An example of a backup trend query is shown below.

\begin{lstlisting}[style=sql,caption={Example backup SQL for a monthly trend question.}]
SELECT strftime('%Y-%m', order_date) AS period,
       SUM(total_amount)             AS revenue
FROM   fact_orders
GROUP  BY period ORDER BY period ASC;
\end{lstlisting}

\subsection{SQL Safety Checks}

Every generated SQL query must pass three checks before it is allowed to run:
\begin{enumerate}[leftmargin=1.5em,itemsep=0pt,topsep=2pt]
\item \textbf{Keyword filter}: the query is scanned for dangerous words like INSERT,
      UPDATE, DELETE, DROP, ALTER, and CREATE. If any are found, the query is rejected.
\item \textbf{Function filter}: specific database functions that could cause harm
      (\texttt{pg\_sleep}, \texttt{dblink}, \texttt{copy}) are blocked.
\item \textbf{Structure check}: the query is parsed to confirm it is a simple SELECT
      or WITH statement. Queries that try to run two statements at once (using a
      semicolon in the middle) are also rejected. A dry-run test is performed
      before the query actually reads any data.
\end{enumerate}
If a query uses \texttt{SELECT *} to fetch all columns, a warning is shown but the query
is still allowed to run.

\subsection{CSV Upload and Auto-Schema Building}

When a user uploads a CSV file, the system processes it in seven steps:
\begin{enumerate}[leftmargin=1.5em,itemsep=0pt,topsep=2pt]
\item \textbf{Read the file} — tries several common text encodings until the file opens.
\item \textbf{Clean column names} — converts all headers to lowercase with underscores
      (for example, ``Product Name'' becomes \texttt{product\_name}).
\item \textbf{Remove bad rows} — drops completely empty rows and exact duplicates.
\item \textbf{Detect data types} — if at least 70\% of values in a column look like
      dates, the column is treated as a date. If at least 90\% look like numbers,
      it is treated as numeric. Currency symbols and percentage signs are stripped first.
\item \textbf{Classify each column} — columns named with endings like \texttt{\_id} or
      \texttt{\_code} are treated as identifiers. Columns with names like
      \texttt{revenue} or \texttt{amount} are measures. Columns with few unique values
      (up to 50, or less than 5\% of all rows) are treated as categories (dimensions).
\item \textbf{Build the schema} — one main fact table is created with all columns,
      plus one small lookup table for each category column.
\item \textbf{Save to the database} — the tables are written to SQLite and all future
      queries are restricted to this uploaded data only.
\end{enumerate}

\subsection{Finding Patterns and Creating Outputs}

\textbf{Pattern detection.} After a query runs, the system looks for four things in the
results: the strongest single value; any unusually high or low values (a value is flagged
as unusual if it is more than 1.75 standard deviations from the average, using at least
5 rows); an overall direction of change over time, shown as a percentage increase or
decrease from first to last value (needs a date column and at least 3 rows); and a
suggestion to try clustering if the results have two or more numeric columns and at
least 12 rows.

\textbf{Chart selection.} The system picks the most suitable chart automatically: a line
chart when the results have a date column, a bar chart when they have a category column, a
scatter chart when they have two numeric columns, or a plain table when none of the above
apply. If the data has too many points for a clear line chart, dates are grouped by month.

\textbf{Export formats.} Results can be downloaded in three formats. Each is generated
independently so a problem with one does not affect the others. The CSV file contains the
full results table. The Excel file has two sheets: one for the data and one for the SQL
and key insights. The PDF report includes a title, the top three findings, the SQL used,
a chart image, and a preview of the first six result rows.

% ─── 7. CONCLUSION ─────────────────────────────────────────
\section{Conclusion and Future Work}

This project shows that a system built from multiple AI agents, with strict safety checks
and smart schema understanding, can reliably answer business questions by writing and
running SQL — without the user needing any database knowledge. The system handled all five
question types correctly, passed all 22 tests, and continued to work in offline mode
using the backup generator.

Planned improvements include: running the AI model locally on the user's own computer
(removing the need for an internet connection); using smarter column-matching to handle
questions with ambiguous wording~\cite{devlin2019bert}; supporting follow-up questions
that build on previous results; testing against the standard Spider SQL
benchmark~\cite{yu2018spider}; adding user accounts for team use; and supporting voice
questions via speech-to-text~\cite{radford2022whisper}.
