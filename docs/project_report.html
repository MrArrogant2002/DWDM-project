1. Abstract

This project presents an Autonomous SQL Agent — an end-to-end agentic AI system that enables non-technical users to perform complex data analysis on their own datasets without writing a single line of SQL. Users upload any CSV file; the system automatically detects column data types, strips currency symbols and percentage suffixes, normalises missing values, and builds a star-schema data warehouse (fact table plus one dimension table per low-cardinality column). Users then ask questions in plain English, and a multi-agent pipeline — consisting of intent detection, planning, schema-aware SQL generation via a large language model (LLM), validation, execution, pattern discovery, and report generation — produces result tables, interactive charts, and downloadable exports in one seamless workflow.

The system is designed for robustness across real-world datasets: it handles five encodings, four field delimiters, currency-prefixed numbers, percentage-suffixed strings, and malformed LLM JSON outputs. When the LLM fails, a schema-aware rule-based fallback always produces a valid result. The project demonstrates that agentic AI pipelines, combined with lightweight local databases, can democratise data analysis for domain experts with no SQL background.

2. Introduction

The exponential growth of data in retail, e-commerce, healthcare, and logistics has created a critical bottleneck: most domain experts who possess the business context to interpret data lack the SQL skills to query it. Traditional solutions — hiring data analysts, building BI dashboards, or using drag-and-drop report builders — are either expensive, rigid, or require significant infrastructure.

Recent advances in large language models, particularly code-specialised models such as Qwen2.5-Coder and Defog's SQLCoder, have made it feasible to translate natural language questions directly into SQL. However, existing text-to-SQL systems typically require pre-defined schemas, clean and normalised data, and expert-curated table metadata — none of which are available when a user simply has a raw CSV file.

This project bridges that gap. The system takes any CSV, automatically engineers a relational star schema from it, builds a SQLite data warehouse, and then accepts natural language questions. A multi-agent AI pipeline interprets the question, generates SQL tailored to the specific schema, validates and executes it safely, and returns results with charts and natural-language summaries — all without the user ever touching a database or writing a query.

Core proposition: Any CSV file → automatic star schema → natural language queries → results, charts, and exports. Zero SQL knowledge required from the user.

3. Objectives

  1. Automatically ingest and profile any CSV file regardless of encoding, delimiter, or column naming convention.
  2. Infer column data types (numeric, datetime, categorical, identifier, free-text) and map them to a normalised star schema.
  3. Build a SQLite data warehouse in-memory with fact and dimension tables, ready for immediate querying.
  4. Accept natural language questions and classify user intent (trend, comparison, anomaly, segmentation, summary).
  5. Generate correct, safe SQLite SELECT queries using an LLM (HuggingFace Inference API) with schema context and business glossary.
  6. Validate all generated SQL for correctness and safety before execution — block DDL, DML, multi-statement, and dangerous functions.
  7. Execute validated queries, detect statistical patterns and anomalies, and generate interactive Plotly charts.
  8. Provide downloadable results in CSV, Excel, and PDF formats.
  9. Ensure graceful degradation: a rule-based fallback generator guarantees results even when the LLM fails.
  10. Deliver all functionality through a clean Streamlit web interface accessible to non-technical users.

4. System Architecture

The system follows a layered architecture with five distinct tiers: the user interface, the orchestration layer, the agent pipeline, the data layer, and the inference layer.

Layer 1 — User Interface
Streamlit web application with two tabs: (a) Upload & Process, (b) Query Warehouse. Handles file upload, progress animation, schema visualisation, query input, result display, and downloads.
Layer 2 — Orchestration (AnalysisOrchestrator)
Coordinates the full analysis pipeline: invokes agents in sequence, manages the retry loop for SQL generation, enforces table-scoping from the active blueprint, and assembles the final AnalysisResponse.
Layer 3 — Agent Pipeline
Seven specialised agents communicate through a shared AgentState object: IntentAgent → PlanningAgent → SQLGenerationAgent → SQLValidationAgent → ExecutionAgent → PatternDiscoveryAgent → ReportAgent.
Layer 4 — Data Layer
CSVIngestor builds the star schema and writes to SQLite via SQLAlchemy. DatabaseManager provides read-only query_dataframe() and explain_query() methods for the analysis path. ExportService writes CSV, XLSX, and PDF artefacts.
Layer 5 — Inference Layer
HuggingFaceSQLGenerator calls the HuggingFace Inference API (Qwen2.5-Coder-7B-Instruct). Falls back to a schema-aware rule-based SQL generator when the API is unavailable or returns malformed output.

5. Implementation Details

5.1 CSV Ingestion Pipeline (csv_ingestion.py)

The ingestion pipeline is the most technically complex component, designed to handle the full diversity of real-world CSV files.

Encoding & Delimiter Detection

The system tries five encodings in order: UTF-8, UTF-8-BOM, Latin-1, CP1252, and ISO-8859-1. Each attempt uses pd.read_csv with sep=None and engine='python' to auto-detect the field delimiter via Python's csv.Sniffer. This handles comma, semicolon, tab, and pipe-delimited files without configuration.

Column Name Sanitisation

All column names are converted to unique SQL-safe snake_case identifiers via _to_snake(). The function strips non-word characters, collapses underscores, guards against SQL keywords (SELECT, FROM, etc.), prefixes digit-leading names with col_, and appends numeric suffixes to resolve duplicates.

Type Coercion

Object-dtype columns go through a three-stage coercion pipeline:

  1. Normalisation: Whitespace stripping, common NA-string replacement (null, N/A, -, --)
  2. Datetime detection: Hit-rate evaluated on non-null values only, using format="mixed" for pandas ≥ 2.0
  3. Numeric detection: Currency symbol stripping ($ € £ ¥ ₹ etc.) and percentage suffix removal before numeric coercion

Column Role Classification

RoleDetection RuleUsed For
datepandas datetime64 dtypeTime-series analysis, trend queries
idName matches regex (_id, _key, phone, zip, etc.)Join keys — excluded from aggregation
measureNumeric dtype, not an IDSUM, AVG, aggregation targets
dimensionString, ≤ 50 unique valuesGROUP BY, filtering, dimension tables
textString, > 50 unique valuesFree-text — kept in fact table only

Star Schema Generation

For each dimension column, a separate lookup table is created (dim_{column_name}) containing unique values with a surrogate integer key. The fact table contains all original columns plus foreign key references. Dimension table names are deduplicated with numeric suffixes if collisions arise.

5.2 Multi-Agent Pipeline (agents.py, orchestrator.py)

IntentAgent

Classifies user intent into one of five categories by keyword matching: trend (monthly, over time), anomaly (unusual, outlier, spike), segmentation (segment, cluster, group), comparison (compare, vs, versus), or summary (default). Also detects whether a plain-English summary is requested.

PlanningAgent

Builds a step-by-step analysis plan (4–6 steps) tailored to the detected intent. This plan is injected into the LLM prompt to guide SQL generation toward the right aggregation strategy.

Schema Context

The orchestrator calls SchemaMetadataService.build_schema_summary() directly, reading column names and types from SQLite's PRAGMA table_info. Table access is scoped to the currently active blueprint via a table filter, preventing queries from leaking across dataset uploads.

Retry Loop

SQL generation runs in a retry loop (default: 3 attempts). On failure, the error message plus the list of valid table and column names is fed back to the LLM as additional context. After all retries, the rule-based fallback is invoked as a final safety net.

5.3 LLM Integration & SQL Generation (model.py)

HuggingFace Inference API

The primary SQL generator calls Qwen2.5-Coder-7B-Instruct via the HuggingFace Inference API using a structured JSON prompt. The prompt includes the full schema context, analysis plan, business glossary, and nine critical SQLite-specific syntax rules. The model is asked to return a JSON object with fields: sql, analysis_goal, tables_used, chart_hint, and needs_summary.

Robustness Features

  • ANSI stripping: Terminal escape codes emitted by some models are stripped before JSON parsing.
  • Markdown fence removal: Code blocks in the response are stripped.
  • Partial JSON extraction: If the response is truncated (token limit), the SQL field is extracted via regex from the incomplete JSON.
  • max_tokens = 2048: Doubled from 1024 to prevent mid-JSON truncation on large schemas.

Rule-Based Fallback

Five hand-crafted SQL templates cover the most common analysis patterns: top-N by dimension, monthly trend (using strftime('%Y-%m', date)), returns by region, segment comparison, and generic aggregation. Column names for each template are resolved by searching the live schema context for type and name hints — never hardcoded defaults.

5.4 SQL Validation & Security (sql_validation.py)

Every generated SQL passes through a three-stage validation gate before execution:

  1. Keyword scan: Blocks INSERT, UPDATE, DELETE, DROP, CREATE, TRUNCATE, EXEC, and other write/DDL keywords.
  2. sqlglot parse: Full AST-level parse with SQLite dialect. Validates syntax and normalises the query.
  3. EXPLAIN QUERY PLAN: Runs the query against the live database engine. Catches invalid table/column names and cross-join explosions without returning any data.

Blocked functions include: pg_sleep, dblink, COPY, and any multi-statement input. The system enforces strict read-only access on the analysis path.

5.5 Analytics & Visualisation (analytics.py, charting.py)

Pattern Discovery

The PatternDiscoveryAgent runs statistical analysis on the query result: computes z-scores to detect numeric outliers, identifies the top and bottom values in each column, and generates contextual follow-up question suggestions based on the detected intent.

Chart Inference

The ChartService infers the appropriate chart type from the result schema and the model's chart_hint:

  • Line chart: When a date/month column is present (time series). Auto-aggregates to monthly buckets if raw unaggregated data is detected (n_unique == n_rows and n_rows > 50).
  • Bar chart: When a categorical column is paired with a numeric column.
  • Scatter chart: When two numeric columns are present.

Plotly Express is used for interactive UI charts. Matplotlib/Agg is used for static PDF export charts.

5.6 User Interface (app/streamlit_app.py)

Upload & Process Tab

Animated progress bar with a step-by-step display of the ingestion pipeline stages. After successful processing, the generated star schema is visualised with colour-coded column pills (measure, dimension, date, id) and dimension table cards.

Query Tab

Five dataset-specific quick-prompt buttons are generated dynamically from the uploaded schema — selecting the most relevant metric, dimension, and date columns using a tiered priority search. Clicking a button fills the text area via Streamlit session state. Results are displayed in a paginated table, an interactive Plotly chart, and a plain-English summary section.

7. Challenges & Solutions

ChallengeRoot CauseSolution
Column names break SQL CSVs with spaces, special chars, SQL keywords, or digit-leading names _to_snake() normalisation with SQL keyword guard and digit prefix
Currency/percent not recognised as numbers Strings like "$1,234.50" or "25%" are object dtype Regex stripping of currency symbols and % before numeric coercion
LLM hallucinates column names Model uses training-data knowledge instead of provided schema On retry: inject explicit column list + table list into error feedback
LLM output truncated mid-JSON max_tokens=1024 too low for complex schemas Raised to 2048; added regex SQL field extraction as last resort
ANSI codes in SQL output HF model emits terminal escape codes (\x1b[4m) inside JSON Regex strip all ANSI sequences before JSON parsing
Spaghetti line chart Fallback trend SQL used raw date column, returning one row per transaction Always use strftime('%Y-%m', date) AS month + SUM() in trend fallback
PyArrow rejects currency regex Backslashes before non-special unicode chars invalid in RE2 engine Removed backslashes: [$€£¥₹...] instead of [\$\€\£...]
Small datasets misclassify columns Dynamic cardinality threshold (5% of n) gave ceiling=2 for 5-row sets Fixed ceiling at 50; dynamic threshold removed
Numeric measures classified as IDs Old cardinality heuristic (n_unique/n > 0.85) flagged price, latitude as IDs Name-based ID detection only using _ID_NAME_RE regex

8.2 Functional Verification

The system was validated against three publicly available retail/e-commerce datasets:

  • Kaggle Supermarket Sales (1,000 rows, 17 columns): Correct detection of date column, sales metric, and six dimension columns. Monthly trend query produced a 12-point line chart.
  • Sample Superstore (9,994 rows, 21 columns): Order date correctly parsed as TIMESTAMP; sales, profit, and discount identified as measures; category, region, and segment as dimensions.
  • Retail Sales Analytics (1,000 rows, 9 columns): No date column — system correctly omitted trend prompts and offered distribution queries instead.

8.3 Key Performance Observations

  • CSV ingestion completes in under 2 seconds for datasets up to 50,000 rows.
  • LLM API response time: 3–8 seconds per request (Qwen2.5-Coder-7B via HF Inference API).
  • Rule-based fallback fires in < 50 ms — no LLM dependency.
  • SQL validation (sqlglot parse + EXPLAIN) adds < 100 ms per attempt.
  • Export generation (CSV + XLSX + PDF) completes in < 1 second for results under 10,000 rows.

9. Conclusion & Future Work

9.1 Conclusion

The Autonomous SQL Agent successfully demonstrates that it is possible to build a production-quality, fully automated data analysis system that requires zero database knowledge from the end user. The combination of automated schema inference, multi-agent orchestration, LLM-powered SQL generation, and robust fallback mechanisms creates a system that is both powerful and resilient.

The project makes several practical contributions: a novel multi-pass column type classifier that handles real-world data quality issues; a schema-aware fallback SQL engine that selects columns by type rather than hardcoded names; and a robustness layer around LLM outputs that handles ANSI codes, truncated JSON, and hallucinated identifiers.

9.2 Future Work

  • Multi-table joins: Automatically generate JOIN queries across fact and dimension tables.
  • Conversational context: Maintain a conversation history so follow-up questions can reference prior results.
  • Fine-tuned model: Fine-tune SQLCoder or a similar model on the generated schema format for higher first-attempt success rates.
  • PostgreSQL support: Extend to cloud-hosted PostgreSQL for datasets too large for SQLite.
  • Streaming results: Use Streamlit's streaming API to show results incrementally.
  • Data quality report: Auto-generate a data quality summary (null percentages, outlier counts, distribution statistics) as a separate download.
  • Voice input: Integrate speech-to-text so users can ask questions by voice.
