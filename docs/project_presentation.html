Problem Statement

The gap between data and insights for non-technical users

  • Most domain experts lack SQL skills to query data they understand better than anyone else
  • Traditional BI tools require pre-built dashboards — inflexible for ad-hoc questions
  • Hiring data analysts is expensive and slow — days to answer one business question
  • Text-to-SQL systems exist, but require clean schemas and expert metadata — not raw CSVs
  • When a user has a CSV file, there is no end-to-end system that takes them from raw data to chart-ready insights with natural language

System Overview

Two modes, one seamless workflow

📂 Mode 1 — CSV Upload

  • User uploads any retail/e-commerce CSV
  • Auto-detect encoding + delimiter
  • Infer column roles (measure / dimension / date / id)
  • Build star schema (fact + N dim tables)
  • Write to SQLite data warehouse
  • Show schema with colour-coded pills

🔍 Mode 2 — Query Warehouse

  • Dataset-specific quick prompt buttons
  • Type any natural language question
  • Multi-agent pipeline: intent → plan → SQL → validate → run
  • Display result table + interactive chart
  • Generate plain-English summary if requested
  • Download CSV, Excel, or PDF

Key Features

  • 5 encodings: UTF-8, UTF-8-BOM, Latin-1, CP1252, ISO-8859-1
  • Auto-detect comma, semicolon, tab, pipe delimiters
  • Currency stripping: $ € £ ¥ ₹ ₩ ...
  • Percentage suffix handling (25% → 25.0)
  • NA-string normalisation (null, N/A, -, --)
  • All-null column auto-drop

Column Role Detection

  • Date — pandas datetime64, ≥70% parse rate
  • ID — name regex: _id, _key, phone, zip, uuid…
  • Measure — numeric, not an ID
  • Dimension — string, ≤50 unique values
  • Text — string, >50 unique values

Multi-Agent Pipeline

Seven specialised agents communicate through a shared AgentState

1 · IntentAgent
Classifies user intent: trend, anomaly, segmentation, comparison, summary. Detects if a plain-English summary is needed.
2 · PlanningAgent
Builds a 4–6 step analysis plan tailored to the detected intent. Plan is injected into the LLM prompt.
3 · SQLGenerationAgent
Calls LLM with schema + plan + glossary. On retry, adds valid table/column names to error feedback.
4 · SQLValidationAgent
Three-stage gate: keyword scan → sqlglot AST parse → EXPLAIN QUERY PLAN. Blocks all write operations.
5 · ExecutionAgent
Runs the validated query via read-only query_dataframe(). Enforces export row limit cap.
6 · PatternDiscovery
Detects statistical outliers, top/bottom values, trend direction, and generates follow-up questions.

LLM Integration & Fallback

Dual-mode generation: LLM API + schema-aware rule engine

🤗 Primary — HuggingFace API

  • Model: Qwen2.5-Coder-7B-Instruct
  • Prompt: schema + plan + 9 SQLite rules
  • Output: JSON with sql, chart_hint, needs_summary
  • max_tokens = 2048 (prevents truncation)
  • ANSI code stripping before JSON parse
  • Partial SQL extraction from truncated output
  • Retry with column list in error feedback

⚡ Fallback — Rule-Based Engine

  • Fires when: no HF token, API error, all retries fail
  • 5 templates: top-N, monthly trend, returns-by-region, segments, generic
  • Columns resolved by type-aware search: TIMESTAMP type → date col; REAL type → metric col
  • Trend SQL always uses strftime('%Y-%m', date) AS month
  • Never uses hardcoded column names

SQL Validation & Security

Three-stage gate — no query runs without passing all checks

  • Stage 1 — Keyword Scan: Blocks INSERT, UPDATE, DELETE, DROP, CREATE, TRUNCATE, EXEC, COPY, pg_sleep, dblink
  • Stage 2 — sqlglot AST Parse: Full abstract syntax tree parsing with SQLite dialect. Validates syntax, normalises identifiers, rejects multi-statement input
  • Stage 3 — EXPLAIN QUERY PLAN: Executes against the live database engine without returning data. Catches invalid table/column references that slip past the AST parser

Read-Only Enforcement

  • Only query_dataframe() and explain_query() exposed to agents
  • execute_sql() and write_dataframe() are write-path only — used only by seed scripts
  • Database URL from env — never from user input

Security Posture

  • No secrets in code — all config from .env
  • LLM tool outputs treated as untrusted — sanitised before use
  • Export row cap (50,000 rows) prevents memory exhaustion
  • Statement timeout on PostgreSQL path

Smart Quick Prompts

Dataset-specific prompts generated from the actual uploaded schema

How It Works

  • Reads measure, dimension, date columns from blueprint
  • Three-tier priority search: e.g., prefers gross_income over unit_price
  • 5 prompts targeting different intents: top-N, trend, compare, anomaly, summary
  • Clicking a button fills the text area via session state
  • Clears automatically when a new file is uploaded

Example — Supermarket Sales

  • Show the top 10 product line by gross income from fact table
  • What is the monthly gross income trend over date?
  • Compare gross income and record count across product line
  • Find unusual patterns in gross income by customer type
  • Summarize the key findings from the fact table
